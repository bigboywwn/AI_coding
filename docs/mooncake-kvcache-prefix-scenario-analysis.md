# KVCache Prefix 复用场景分析

## 目的

本文档用于固化当前已经对齐的 `kvcache` 读写场景结论，作为后续硬件与软件设计讨论的统一锚点。

这里区分两件事：

- 目标系统想解决的场景
- SGLang 当前实现给出的工程启发

SGLang 是参考实现，不直接等价于目标场景定义。

## 分析范围

本文档的场景抽象先聚焦 `prefix 复用 KV`，后半部分在此基础上展开 `PD 分离` 下的硬件分析。

不展开通用冷数据归档。

关注 3 条核心流量：

1. `prefix 复用读`：从存储系统读取可复用前缀，并装载到本轮执行层
2. `prefill 新增写`：本轮输入产生的新 KV 写入可复用层
3. `decode 新增写`：本轮输出产生的新 KV 写入可复用层

## 核心对象

系统维护的不是“整条对话的单个大对象”，而是：

- `已发布前缀`：已经稳定、下一轮可以直接复用的前缀页/段
- `活跃尾部`：当前轮还在增长、可能继续追加、尚未完成发布的最新部分

因此更自然的抽象是：

- 逻辑层：前缀命名空间、会话/租户隔离、可见性边界
- 物理层：页/段化存储、切片、分布式放置、分层缓存

## 场景一：Prefix 复用读

### 定义

`prefix 复用读` 是将“已发布前缀”回装到本轮执行层的过程。

### 关键特征

- 读取对象必须是已发布、已 seal 的稳定前缀
- 读取目标是本轮执行层，不应绑定为某一种固定中转路径
- 读取可以在请求入队、排队、调度阶段提前触发
- 读取通常是大块、有序、按页/段的批量回装

### 路径定义

理论上允许多种物理路径：

- `storage -> HBM`
- `storage -> DDR -> HBM`
- `storage -> peer GPU/HBM`
- `storage -> host staging -> GPU`

因此场景定义不应被 SGLang 当前的 `storage -> host -> HBM` 路径限制。

### 系统约束

- 必须有明确版本/可见性边界，不能读到半发布数据
- 必须支持 admission-time prefetch
- 必须支持大块回装，而不是被迫退化为细粒度随机小读

## 场景二：Prefill 新增写

### 定义

`prefill 写` 是将“本轮输入对应的稳定前缀增量”发布到可复用层的过程。

### 关键特征

- 输入内容在 prefill 时已经确定
- 不涉及 decode 那种逐 token 回滚问题
- 只要某个页/段计算完成，就具备被发布的资格

### 场景结论

`prefill 写` 不应理解为“整轮结束后一次性落盘”，而应定义为：

- 渐进式发布
- 按页/段 seal 后发布
- 可边 prefill 边写入可复用层

### 系统约束

- 必须支持分段提交，而不是只能整对象提交
- 必须支持边计算边发布，降低下一轮启动等待
- 必须支持按价值筛选，避免把不会进入下轮上下文的临时内容都写进去

## 场景三：Decode 新增写

### 定义

`decode 写` 是将“当前轮输出产生的新前缀部分”发布到可复用层的过程。

### 关键特征

- 数据是持续追加的
- 存在 speculative / rollback / unaccepted token
- 最新部分处于活跃尾部状态

### 场景结论

`decode 写` 不应被定义成“每 token 必须远端同步写”，也不应被限制成“只能最后整轮一次写”。

更准确的定义是：

- 活跃尾部的增量发布流量
- 以 committed / visible output 为发布边界
- 发布粒度可以是 request-finish、page-seal、window-seal

### 系统约束

- speculative 或未提交 token 不应进入可复用发布层
- 需要把“本地活跃延续”和“远端可复用发布”分开
- decode 主路径不应被远端写入策略强绑定

## 本地延续与远端发布要分开

这是这轮分析最重要的修正点之一。

同一会话、同一推理节点、热 KV 仍在本地时，后续轮次可以优先直接续用本地活跃 KV。

因此：

- `same-session local continuation` 是一个独立旁路场景
- 它会显著降低 `decode 写 -> 远端发布 -> 下轮再读回` 的必要性
- 后续设计时不能把“本地续用”和“远端复用”混成一个机制

## SGLang 给出的校正点

SGLang 不是场景定义本身，但它帮助修正了两个重要误区：

1. `prefill` 产生的稳定前缀可以提前进入可复用结构，不必等整轮完成
2. 标准 `decode` 路径通常不会每个 step 都写 prefix cache，真正的远端发布边界要比 token 粒度更粗

同时，SGLang 的 HiCache 说明：

- 远端复用路径可以做成分层缓存
- 存储读取可以提前触发
- 外存对象更适合页/段化，而不是整条 prefix 大对象

## 后续硬件分析的输入假设

后续讨论硬件时，以如下抽象为准：

1. 读路径是“已发布前缀回装到执行层”，允许直通 `HBM`
2. `prefill 写` 是稳定输入前缀的渐进式发布
3. `decode 写` 是活跃尾部在提交边界上的增量发布
4. 本地续用与远端发布是两个不同层次的问题
5. 系统真实承载的是页/段化前缀，而不是单一整对象
6. 为了简化讨论，`TP/EP/PP` 统一视为只走 `plane1`
7. 在当前简化假设下，`P-D`、`S-P`、`P-S`、`D-S` 的外部数据流量统一接入 `plane2`
8. `plane3` 不再是必选主平面，而是面向更大规模与更强隔离需求的可选扩展平面

## 一句话总结

这个系统本质上是一个“面向多轮对话 prefix 复用的分层、分段、按提交边界发布的 KVCache 数据系统”。

## 场景分析里的系统类特性

这一节只描述系统语义和软硬配合约束，不把它们当作硬件特性。

### Prefix 复用读的系统类特性

- 需要明确的 `published / visible` 边界，防止读到半发布数据
- 需要 admission-time prefetch 和调度前预取能力
- 需要稳定的前缀命名空间与版本/可见性管理
- 需要页/段级对象管理，避免退化为随机小对象读

### Prefill 写的系统类特性

- 需要页/段 `seal` 后的渐进式发布语义
- 需要分段提交，而不是只能整对象提交
- 需要价值筛选，避免把不会进入下轮上下文的内容写入可复用层
- 需要发布后尽快变为可见，降低下一轮启动等待

### Decode 写的系统类特性

- 需要把 `active tail` 和 `published prefix` 明确区分
- 需要以 `committed / visible output` 作为发布边界
- 需要支持 `request-finish / page-seal / window-seal` 等多种发布粒度
- 需要把本地续用与远端发布拆成两个独立机制

### PD 直传的系统类特性

- 需要 `P -> D` 的 bootstrap / ownership transfer 语义
- 需要传递页/段索引、状态索引、会话标识、并行布局等控制信息
- 需要显式的 ready / ack / timeout / retry 状态机
- 需要在并行布局不一致时支持 KV 的逻辑重映射描述

## PD 分离下的硬件分析输入

后续硬件设计不再假设统一推理节点，而是显式区分：

- `P 节点`：Prefill 节点
- `D 节点`：Decode 节点
- `S 节点`：KVCache 存储节点

在 `PD 分离` 下，Prefix 复用相关的 3 条核心流量需要重新理解：

1. `Prefix 复用读`
   - 主要消费方通常是 `P 节点`
   - 目标是在 prefill 启动前，把已发布前缀回装到执行层
2. `Prefill 新增写`
   - 主要生产方是 `P 节点`
   - 目标是把本轮输入对应的稳定前缀增量发布到可复用层
3. `Decode 新增写`
   - 主要生产方是 `D 节点`
   - 目标是把本轮输出对应的已提交前缀增量发布到可复用层

此外，`PD 分离` 还引入了一条额外但非常关键的流量：

4. `PD 交接`
   - 由 `P 节点` 产生、由 `D 节点` 消费
   - 用于将当前请求从 prefill 阶段平滑切换到 decode 阶段

因此，硬件不能按“单类 KV 访问”设计，而要按：

- `P -> S` 写
- `S -> P` 读
- `P -> D` 或 `P -> S -> D` 交接
- `D -> S` 写

这四类方向性流量联合设计。

## PD 分离下的节点角色

### P 节点

`P 节点` 是高吞吐、重计算、阶段性大块写入的节点。

它的主要硬件诉求：

- 强算力和高 HBM 带宽，用于高吞吐 prefill
- 强前缀回装能力，用于把已发布前缀快速拉回执行层
- 强阶段性写出能力，用于把稳定 prefill 前缀和 PD 交接数据快速发布出去

因此，`P 节点` 更偏向：

- 计算密集
- 大块读写
- 突发式网络出口

### D 节点

`D 节点` 是低抖动、时延敏感、长期持有活跃尾部的节点。

它的主要硬件诉求：

- decode 主路径低抖动
- 较强的本地热层承载能力，用于保留活跃尾部
- 足够的外部数据面资源，避免后台流量冲击 decode 主路径

因此，`D 节点` 更偏向：

- 时延敏感
- 热数据保留
- 背景写出不干扰主路径

### S 节点

`S 节点` 是读写混合、强带宽、分层化的外部 KV 服务节点。

它同时面对：

- `P 节点` 的大块写入
- `P 节点` 的前缀回装读取
- `D 节点` 的背景写入

所以 `S 节点` 的设计重点不是单纯容量，而是：

- NIC 带宽
- DDR 带宽
- SSD 带宽
- 多流量并发隔离

## PD 分离下的网络平面设计

在当前讨论里，继续采用 3 个逻辑网络平面。

### 平面 1：节点内高带宽平面

典型对应 `NVLink / NVSwitch`。

职责：

- `P-P` 的 `TP/EP/PP`
- `D-D` 的 `TP/EP/PP`
- 节点内 GPU 间 KV 重分布
- 从目标 GPU 汇聚或拆分待发送的页/段
- 将回装后的前缀分发到本地需要消费的 GPU

设计原则：

- 在当前简化假设下，它同时承载模型并行热路径与节点内 KV 整理
- 它决定了远端读写前后的本地整理成本
- 如果节点内平面不足，远端路径再快也会被本地分发拖慢

### 平面 2：PD 交接平面

典型对应 `CX NIC` 承载的外部数据通信平面。

职责：

- `P -> D` 的控制/元数据交接
- `P -> D` 的 KVCache 直传
- `S -> P` 的 prefix 回装
- `P -> S` 的 prefill 渐进发布
- `D -> S` 的 decode 增量发布

设计原则：

- 在当前简化假设下，它不承担 `TP/EP/PP`
- 它应成为所有节点外数据流量的统一主平面
- `P-D` 交接、`S-P` 回装、`P-S` 发布、`D-S` 发布都需要在这一个平面上受控共存

### 平面 3：可选扩展平面

典型对应 `DPU / 独立 NIC / 独立 rail` 承载的可选扩展数据平面。

职责：

- 在规模扩大后进一步拆分 `P-D` 与存储流量
- 在需要更强隔离时承接部分外部 KV 流量
- 在需要卸载或扩展时提供额外网络与数据面能力

设计原则：

- 它不是当前简化方案下的必选主平面
- 它的价值在于规模扩展、物理隔离和卸载增强
- 是否部署取决于带宽规模和隔离要求

## 3 个平面的推荐分工

如果按优先级排序：

- 平面 1 负责 `TP/EP/PP` 与节点内聚合和分发
- 平面 2 承载全部外部数据流量
- 平面 3 作为可选扩展与隔离平面

这意味着：

- `prefix 复用读` 的主远端路径走平面 2
- `prefill 写` 的主远端路径走平面 2
- `decode 写` 的主远端路径走平面 2
- `PD 交接` 的主路径走平面 2
- 平面 3 只在规模或隔离要求提升时再引入

## PD 分离下的关键硬件约束

### 1. P 与 D 的硬件优化方向不能相同

`P 节点` 更需要：

- 高 prefill 吞吐
- 强突发读能力
- 强突发写能力

`D 节点` 更需要：

- decode 低尾时延
- 较大的本地热缓存承载空间
- 背景写出不干扰 decode

因此不建议把 `P` 和 `D` 完全做成同构节点后再靠软件区分角色。

### 2. S 节点首先要按带宽平衡设计

在 `PD 分离` 下，`S 节点` 往往同时承受：

- 来自 `P` 的入口写带宽
- 发往 `P` 或 `D` 的出口读带宽
- 来自 `D` 的背景写带宽

如果采用传统路径：

- `NIC -> DDR -> SSD`
- `SSD -> DDR -> NIC`

那么到 `800GbE` 量级后，DDR 带宽很容易先成为瓶颈。

因此 `S 节点` 设计时，必须优先匹配：

- 外部网络双向总带宽
- 主机 DDR 实际可持续带宽
- SSD 并行度与尾延迟稳定性

在此基础上，如果希望进一步降低 host DDR 的带宽压力，`S 节点` 还应优先考虑两类增强型硬件能力：

- `CMB/P2P` 友好能力
  - SSD 支持可用于中转的 `CMB`
  - 平台支持 `PCIe P2P DMA`
  - `NIC` 与 `NVMe` 尽量处于同一 `PCIe switch` 或同一局部拓扑域，避免跨 `root complex`
- `DDIO/L3` 友好能力
  - CPU 支持 `DDIO` 或等价的 I/O 到 LLC 注入能力
  - 具备较大的共享 `L3/LLC`
  - `NIC`、`NVMe`、CPU socket 保持严格同 NUMA/同 socket 局部性

其中：

- `CMB/P2P` 是更强的一类能力，目标是直接减少对 host DDR 主路径的依赖
- `DDIO/L3` 更适合承接一部分短驻留、受控的小窗口中转流量，不能替代 `CMB/P2P`

### 3. Prefix 回装路径应允许直通 HBM

从场景定义上，`prefix 复用读` 不应被限制为 `storage -> host -> HBM`。

如果硬件允许，优先路径应支持：

- `storage -> HBM`
- `storage -> peer GPU/HBM`

做不到时，再退化为：

- `storage -> DDR -> HBM`

因此推理节点和存储节点在硬件上应具备 `GPU` 作为远端数据落点的能力，例如合适的 `GPU-NIC` 拓扑和 `GPUDirect RDMA` 类能力。

### 4. Decode 写不应占用 decode 主路径预算

`D 节点` 的活跃尾部写出，必须在硬件上具备背景化条件：

- 足够的外部数据面带宽
- 独立 DMA/队列
- 明确优先级隔离

否则 decode 的低时延目标会被远端发布流量直接破坏。

## PD 分离下的第一轮硬件结论

1. `P 节点`、`D 节点`、`S 节点` 应按不同目标分别优化，而不是统一配置。
2. `plane1` 是 `TP/EP/PP` 的唯一平面，因此其规模和二分带宽直接决定模型并行上限。
3. `plane2` 是统一外部数据平面，因此其总带宽、优先级隔离与 `GPU-NIC` 亲和直接决定外部数据路径能力上限。
4. `S 节点` 的首要设计问题是 `NIC/DDR/SSD` 带宽平衡；若要进一步降低 host DDR 压力，应优先具备 `CMB/P2P` 友好拓扑，并以 `DDIO/L3` 作为辅助增强能力。
5. `Prefix 复用读` 应优先支持 `storage -> HBM` 直通能力。
6. `Decode 写` 必须被设计成后台化、低干扰流量。
7. `plane3` 是可选增强项，用于更大规模下的额外隔离或卸载扩展。

## PD 分离下的硬件完整特性结论

这一节只保留纯硬件特性，不再写系统/软件配合语义。

### 全部流量与硬件特性对照表

| 流量 | 方向 | 主平面 | 流量属性 | 必须的硬件特性 |
| --- | --- | --- | --- | --- |
| 本地热续用 | 节点内 | `plane1` | 最低时延、本地热数据 | 足够的 `HBM` 容量与带宽；高双向带宽的 `NVLink/NVSwitch` 节点内互联 |
| `P-P` 模型并行 | `P <-> P` | `plane1` | 大带宽、长持续、同步型 | 足够大的 `NVLink/NVSwitch` 域；高二分带宽；`TP/EP/PP` 工作集应尽量闭合在同一 `plane1` 域内 |
| `D-D` 模型并行 | `D <-> D` | `plane1` | 高频、小步、对 `p99` 极敏感 | `NVLink` 域内尽量闭合；高二分带宽；`plane1` 的低抖动特性直接决定 decode 模型并行稳定性 |
| `P-D` 控制/元数据 | `P <-> D` | `plane2` | 小消息、关键控制流 | `CX` 平面具备高优先级队列、低时延交换、低超卖比 |
| `P-D` KV 直传 | `P -> D` | `plane2` | 突发大流量、阶段切换关键路径 | `CX` 平面具备足够带宽余量；`GPU-NIC` 拓扑亲和；`GPUDirect RDMA`；多队列 NIC；交换机支持优先级隔离 |
| Prefix 复用读 | `S -> P` | `plane2` | 大块顺序读、快启动 | `plane2` 具备足够读带宽余量；存储侧高读带宽；`P` 侧 `GPU-NIC` 亲和；支持 `GPU` 作为 RDMA 目标内存 |
| Prefill 写 | `P -> S` | `plane2` | 中大块持续写 | `plane2` 具备持续写带宽；存储侧高写带宽；足够的 PCIe 与 DDR 带宽承接入口流量 |
| Decode 写 | `D -> S` | `plane2` | 后台写、不可干扰 decode | `plane2` 具备独立队列与优先级隔离；不能挤占 decode 主路径对应的关键计算与搬运资源；存储侧稳定入口带宽 |

### 网络平面的硬件特性表

| 平面 | 定位 | 必须的硬件特性 |
| --- | --- | --- |
| `plane1` | `NVLink/NVSwitch` 超平面 | 大规模 `NVLink/NVSwitch` 域；足够高的节点内二分带宽；支持 `P/D` 节点内多 GPU 高速互联；能够承载全部 `TP/EP/PP` |
| `plane2` | `CX` 外部数据平面 | 高带宽 `CX` NIC；低时延无阻塞或低超卖交换；多队列/多优先级；硬件拥塞控制；支持 `GPUDirect RDMA`；支持按 GPU 绑定 NIC；能够同时承载 `P-D`、`S-P`、`P-S`、`D-S` |
| `plane3` | 可选扩展平面 | 可选的独立 `CX/DPU` 数据面；与 `plane2` 物理或 rail 级解耦；用于更大规模下的额外隔离、卸载或扩展 |

### 节点的硬件特性表

| 节点 | 核心职责 | 必须的硬件特性 |
| --- | --- | --- |
| `P` 节点 | prefill 计算、prefix 复用读消费、prefill 写生产、`P-D` 直传发送 | 高算力 GPU；高 `HBM` 带宽；强 `NVLink/NVSwitch` 域内互联；`plane2` 高带宽 `CX` 口；`GPU-NIC` 亲和；`GPUDirect RDMA` |
| `D` 节点 | decode 计算、活跃尾部驻留、`P-D` 直传接收、decode 写生产 | 高算力 GPU；更强调低抖动的 `HBM` 保留空间；强 `NVLink/NVSwitch` 域内互联；`plane2` 高带宽低时延交接口；`GPU-NIC` 亲和；`GPUDirect RDMA` |
| `S` 节点 | prefix 读供给、prefill/decode 写接收 | 高带宽 `plane2` NIC；高 DDR 带宽；足够 PCIe lane/交换带宽；足够 NVMe 并行度与顺序读写带宽；NUMA 对称与 NIC/SSD 本地性良好；优选支持 `CMB/P2P` 友好拓扑；支持 `DDIO` 或等价 I/O 到 `LLC` 的能力 |

### 存储节点 `S` 的关键硬件约束

| 约束项 | 必须的硬件特性 |
| --- | --- |
| 外部网络 | 足够的 `plane2` NIC 总带宽，且支持双向高吞吐 |
| 主机内存 | 高 DDR 带宽，能覆盖 `NIC <-> DDR <-> SSD` 路径放大 |
| PCIe | NIC 与 NVMe 总吞吐不能被 PCIe/PCIe switch 卡住；优选支持 `PCIe P2P DMA`，并保证 `NIC` 与 `NVMe` 处在适合 `P2P` 的局部拓扑域 |
| SSD | 足够的盘数、总顺序带宽、稳定尾延迟、合适耐久度 |
| 拓扑 | NIC、CPU、DDR、SSD 的 NUMA 绑定合理，避免跨 socket 带宽损失 |
| `CMB/P2P` | SSD 侧优选支持 `CMB`；平台优选支持 `CMB/P2P` 直通所需的 `PCIe` 能力与局部交换拓扑 |
| `LLC/DDIO` | CPU 平台优选支持 `DDIO` 或等价能力；具备较大的共享 `LLC/L3`，可承接一部分短驻留中转流量 |
