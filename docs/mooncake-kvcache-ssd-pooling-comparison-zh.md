# KVCache SSD 池化方案对比：传统分布式文件/对象存储 vs MoonCake + NoF SSD

## 1. 背景

KVCache 池化的核心目标，不是单纯把数据写到 SSD 上，而是以更低成本扩展有效缓存容量，同时尽量维持在线推理场景下的吞吐稳定性和服务连续性。

围绕这个目标，业界常见的两类思路是：

- 基于分布式文件系统或对象存储构建 KVCache 的 SSD 池化层。
- 基于 MoonCake 的内存池 + NoF SSD 池构建专项化的多级 KVCache 池化层。

这两条路线在设计目标、IO 语义、可靠性边界和性能路径上有本质差异。本文从 KVCache 自身的 IO 特征出发，对两类方案进行对比，并给出更适合该场景的结论。

## 2. KVCache IO 特征，与传统文件/对象存储 IO 的区别

### 2.1 总体判断

KVCache 更接近“高性能分层缓存 IO”，而不是“传统持久化存储 IO”。

它服务于在线推理过程，具有短生命周期、可回源、可丢失、强时效性和高吞吐/低尾延迟敏感等特点。传统文件系统和对象存储则更强调持久化、通用性、可靠性和共享访问。

### 2.2 对比表：KVCache IO vs 传统文件/对象存储 IO

| 维度 | KVCache IO 特征 | 传统文件/对象存储 IO 特征 | 差异总结 |
| --- | --- | --- | --- |
| IO 大小 | 常表现为块级、chunk 级、分段式访问；粒度受 token block、layer、head 等组织方式影响 | 文件系统偏页/块/文件语义；对象存储偏整个对象或大对象分片 | KVCache 更偏中小块、高频块访问 |
| IO 生命周期 | 短生命周期，常与单次会话、请求窗口、上下文复用周期绑定 | 生命周期更长，面向持久保存、归档和共享访问 | KVCache 是短命数据，传统存储是长命数据 |
| IO 可变性 | 写后读多，更新模式有限；对象生成后往往很快只读并随后过期 | 文件常有覆盖写、追加写、随机修改；对象常为整体覆盖 | KVCache 更接近写一次、读若干次、很快过期 |
| Durability 要求 | 低，允许丢失；重点是服务连续性，不是永久保存 | 高，强调副本、持久化、恢复和一致性 | KVCache 是缓存语义，传统存储是资产数据语义 |
| Metadata 特征 | 关注 session、block、热度、层级、副本状态、租约等运行时信息 | 关注路径、对象键、目录、权限、版本、mtime 等存储管理信息 | KVCache 元数据偏运行时调度，传统元数据偏存储管理 |
| 写放大敏感度 | 非常高；额外 copy、日志、副本都会直接伤害收益 | 较能接受为可靠性付出的写放大 | KVCache 不希望为强持久化语义支付额外写放大 |
| 读放大敏感度 | 对 tail latency 极度敏感；额外 hop、协议封装都会伤害读路径 | 一般可接受一定读放大来换通用性和可靠性 | KVCache 对额外读路径更敏感 |
| 访问模式 | 热点明显，冷热分层价值高；prefill 写入与 decode 读取具有明显阶段性 | 更通用，顺序/随机/批量/归档混合 | KVCache 天然适合多级缓存和分层 |
| 并发特征 | 高并发、短事务、强时效 | 高并发也常见，但更偏吞吐和共享访问 | KVCache 更强调在线时延和稳定性 |
| 数据价值密度 | 单个对象价值低，丢失代价通常是重算 | 单个对象价值通常更高，丢失代价大 | KVCache 更适合可丢失设计 |
| 一致性要求 | 相对较弱，更关注可读可用和副本状态 | 通常需要更清晰的一致性和持久化语义 | KVCache 不需要过重一致性负担 |
| 命中/淘汰机制 | 是系统核心能力，直接决定收益 | 传统存储通常不以命中/淘汰为核心 | KVCache 本质上是缓存管理问题 |
| 回源语义 | miss 后通常可由上层重算或回源恢复 | 默认底层应保存原始数据 | KVCache 可以接受 miss，传统存储通常不能 |
| 资源目标 | 以较低成本扩大有效缓存容量 | 提供稳定持久化存储池 | 两者目标函数不同 |

### 2.3 小结

KVCache 的问题本质上不是“如何把数据持久化到 SSD”，而是“如何围绕缓存语义、高时效访问和可退化服务，构建一套更低成本的分层对象池”。这决定了它天然更适合分层缓存系统，而不是传统通用存储系统。

## 3. 方案一：基于分布式文件/对象存储的 KVCache SSD 池化

### 3.1 方案描述

该方案通常基于以下基础设施：

- 分布式文件系统，如 CephFS、Lustre、JuiceFS 等
- 分布式对象存储，如 S3、MinIO、Ceph RGW、OSS 等

系统把 KVCache 作为文件或对象进行组织，并借助底层存储的容量扩展能力实现 SSD 池化。

### 3.2 主要优点

- 生态成熟，运维经验丰富。
- 统一的数据副本和持久化能力强。
- 容量扩展能力成熟。
- 接口标准化，容易复用已有系统。
- 对长生命周期和共享访问场景友好。

### 3.3 主要缺点

- IO 路径通常更长，协议栈和元数据路径更重。
- 为持久化和一致性付出的开销，并不完全匹配 KVCache 场景。
- 对中小块、高频、短生命周期对象的适配性较差。
- 尾延迟更难控制。
- 淘汰、热度分层和退化能力通常不是原生核心能力。

## 4. 方案二：基于 MoonCake + NoF SSD 的 KVCache 池化

### 4.1 方案描述

该方案以 MoonCake Store 为核心控制面，以内存池承接热点 KVCache，以 NoF SSD 池承接温数据和容量扩展层。系统围绕对象副本、分层访问、受控淘汰、故障隔离和退化运行进行设计。

### 4.2 主要优点

- 更贴合 KVCache 的负载特征和生命周期特征。
- 可构建内存优先、NoF 回退的更短 IO 路径。
- 更容易围绕缓存收益而不是持久化语义优化系统。
- 淘汰、分层、退化和回源语义更加自然。
- 更适合以服务连续性为核心的可靠性目标。

### 4.3 主要缺点

- 通用性较弱，更偏专项化系统。
- 对 NoF / SPDK / 网络拓扑和调优能力要求更高。
- 相比传统存储生态，工程成熟度和运维经验积累通常更少。
- 系统内部状态管理会更复杂，尤其是分层、副本、淘汰和故障路径的联动。

### 5.2 关键维度展开

#### 1. 设计目标

传统分布式文件/对象存储的设计目标是通用持久化存储，而 MoonCake + NoF SSD 的设计目标是高性能、多级、可退化的 KVCache 对象池。从目标函数上看，后者更贴近 KVCache 场景。

#### 2. IO 路径与性能

传统方案通常会引入更长的元数据路径、协议栈和对象封装路径，更容易产生额外尾延迟。MoonCake + NoF SSD 可以构建“内存命中优先、NoF 回退”的专项路径，更容易在在线推理场景中控制 tail latency 和吞吐抖动。

#### 3. 容量控制与淘汰

KVCache 池化不是单纯存储问题，淘汰能力是核心能力。传统文件/对象存储并不是围绕淘汰和热点管理设计的，而 MoonCake 方案天然可以把淘汰作为容量控制和收益控制的一部分。

#### 4. 可靠性边界

传统方案强调“数据不能丢”，MoonCake 方案强调“服务不能断”。对于 KVCache 这种可回源、可重算的数据类型，后者更符合业务目标。

#### 5. 成本模型

传统存储经常为强语义支付额外成本，例如副本、纠删码、日志、复杂元数据路径。MoonCake + NoF SSD 更容易把资源投入到“有效缓存容量、分层收益和服务连续性”这些真正与 KVCache 价值相关的地方。

## 6. 结合 KVCache 特征与方案差异，选择更优方案

### 6.1 结论先行

如果目标场景是**在线推理下的 KVCache SSD 池化**，更优方案是：

**MoonCake + NoF SSD**

### 6.2 原因总结

原因主要有以下几点：

- KVCache 本质上是缓存，不是长期持久化数据。
- KVCache 生命周期短，可回源，可淘汰，可退化。
- KVCache 对延迟、尾延迟和吞吐波动非常敏感。
- KVCache 的收益高度依赖多级池化、热点管理和受控淘汰。
- KVCache 的可靠性目标更接近“服务连续性”而不是“数据强恢复”。

MoonCake + NoF SSD 在这些维度上，与 KVCache 的本质要求是一致的；而传统文件/对象存储虽然可行，但更多是把 KVCache 作为普通数据去保存，目标函数并不完全一致。

### 6.3 什么情况下传统方案仍然值得考虑

传统方案在以下场景仍然可能是合理选择：

- 团队已经有非常成熟的对象/文件存储基础设施，希望快速复用。
- 对性能要求不是极致优先，而更看重系统通用性和成熟度。
- KVCache 的角色更偏共享中间结果，而不是强在线时效缓存。
- 团队对 NoF / SPDK / 分层缓存系统的专项维护能力不足。

换句话说：

- 如果目标是“通用、成熟、工程省心”，传统方案更友好。
- 如果目标是“更贴合 KVCache、本质更匹配、性能和收益更优”，MoonCake + NoF SSD 更合适。

## 7. 最终结论

### 7.1 结论表

| 问题 | 结论 |
| --- | --- |
| KVCache IO 更像哪类 workload | 更像高性能分层缓存 IO，而不是传统持久化存储 IO |
| 传统文件/对象存储能否用于 KVCache SSD 池化 | 可以，但通常不是最优匹配 |
| MoonCake + NoF SSD 是否更贴合 KVCache | 是，更贴合其生命周期、性能目标和可靠性边界 |
| 哪个方案更适合在线推理 KVCache 池化 | **MoonCake + NoF SSD 更优** |
| 传统方案的主要优势 | 成熟度、通用性、运维经验和生态复用 |
| MoonCake 方案的主要优势 | 语义匹配、分层和淘汰更自然、IO 路径更短、服务连续性目标更匹配 |

### 7.2 最终判断

对于 KVCache SSD 池化问题，最优方案不是“最强持久化的存储方案”，而是“最符合缓存语义的分层池化方案”。

因此，从 KVCache 的 IO 特征、生命周期特征、可靠性边界和成本收益目标来看：

**MoonCake + NoF SSD 是更适合该场景的方案。**
